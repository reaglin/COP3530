Quicksort

▶

Quicksort is one of the most widely used and practically efficient sorting algorithms ever devised. Introduced by Tony Hoare in 1959, it operates on the principle of divide and conquer: select a pivot element, rearrange the array so that all elements smaller than the pivot sit to its left and all elements larger sit to its right, then recursively apply the same process to each of the two resulting sub-arrays. Unlike merge sort, which divides the array mechanically in half and does its main work during the merge step, quicksort does its main work before recursing — in the partition step — and requires no additional array allocation to do so. The result is an algorithm that is elegant, in-place, and extraordinarily fast in practice.

Understanding quicksort deeply means understanding four interlocking ideas: how partitioning physically rearranges elements, how pivot choice shapes performance, what the algorithm's efficiency guarantees actually are, and how to translate the concept into working code. Each of these is explored in full below.

Partitioning Strategy

Partitioning is the engine of quicksort. Given a sub-array and a chosen pivot value, the partition operation scans the array and swaps elements so that every value less than the pivot ends up to its left and every value greater ends up to its right. When partitioning is complete, the pivot is placed at its final, correct position in the sorted array and never needs to move again. This is a crucial insight: after a single partition, at least one element — the pivot — is permanently in its sorted position.

The operation is performed in-place. No second array is allocated; elements are swapped within the original array using only a constant amount of extra memory for index variables. This is one of the reasons quicksort is memory-efficient and cache-friendly compared to algorithms that allocate auxiliary arrays.

The most common partitioning scheme is the Lomuto partition scheme, which uses a single pointer i to track the boundary between elements known to be smaller than the pivot and everything else. A second pointer j scans forward through the array. Whenever j finds an element smaller than or equal to the pivot, i is incremented and the elements at i and j are swapped. After the scan, the pivot (typically stored at the last position) is swapped into position i + 1, which is its final sorted location.

The Hoare partition scheme — the original — uses two pointers that start at opposite ends of the array and move toward each other. The left pointer advances until it finds an element greater than or equal to the pivot; the right pointer retreats until it finds an element less than or equal to the pivot. The two elements are swapped, and the process continues until the pointers cross. Hoare's scheme typically performs fewer swaps on average than Lomuto's, though it is slightly more complex to implement correctly.

To make this concrete, consider partitioning the array [3, 6, 8, 10, 1, 2, 1] using Lomuto's scheme with the last element (1) as the pivot:

  • Initial state: [3, 6, 8, 10, 1, 2, 1], pivot = 1, boundary pointer i = -1
  • j=0: element 3 is not ≤ 1, no swap.
  • j=1: element 6 is not ≤ 1, no swap.
  • j=2: element 8 is not ≤ 1, no swap.
  • j=3: element 10 is not ≤ 1, no swap.
  • j=4: element 1 ≤ 1 → increment i to 0, swap positions 0 and 4: [1, 6, 8, 10, 3, 2, 1]
  • j=5: element 2 is not ≤ 1, no swap.
  • Final swap: place pivot at i+1 = 1: [1, 1, 8, 10, 3, 2, 6]

The pivot value 1 is now at index 1, which is its correct sorted position. Everything to its left is ≤ 1 and everything to its right is ≥ 1. Each recursive call then processes a strictly smaller sub-array — [1] on the left and [8, 10, 3, 2, 6] on the right — until sub-arrays of zero or one element are reached, at which point they are trivially sorted.

Pivot Selection Considerations

The pivot is not just an implementation detail — it is the single biggest determinant of quicksort's performance on any given input. The choice of pivot controls how evenly the array is split at each recursive step, which in turn controls the depth of the recursion tree and thus the total work done.

First or last element as pivot is the simplest strategy to code. The pivot is just arr[low] or arr[high]. Unfortunately, this choice is catastrophically bad on arrays that are already sorted or reverse-sorted — two very common real-world inputs. On a sorted array of n elements, choosing the last element as pivot means every partition step places zero elements to the right and all remaining elements to the left, producing a recursion tree of depth n instead of log n. This is the worst-case O(n²) scenario.

Random pivot selection addresses this by choosing the pivot uniformly at random from the current sub-array. Before partitioning, the randomly chosen element is swapped with the last element, and the standard Lomuto scheme proceeds as normal. Because the choice is random, no adversarial input can reliably trigger worst-case behavior. The expected number of comparisons over all inputs becomes O(n log n) with very high probability. This is the most common improvement applied in practice.

Median-of-three is a practical compromise that examines three candidate elements — typically the first, middle, and last elements of the sub-array — and selects the median value as the pivot. For example, if those three values are 8, 3, and 6, the median is 6, which becomes the pivot. This strategy tends to produce more balanced partitions than a fixed-position strategy, handles already-sorted arrays well, and adds only a tiny constant overhead (three comparisons). Many real-world implementations of quicksort, including those in standard libraries, use median-of-three or a variant of it.

It is important to understand that no pivot strategy can guarantee optimal partitioning for every possible input. A median-of-three strategy can still be fooled by specially constructed inputs (the "median-of-three killer"). Random selection is the most robust because its worst case depends on chance rather than input structure, making it essentially impossible to trigger reliably. The table below summarizes the tradeoffs:

Pivot Strategy Implementation Complexity Worst-Case Input Practical Use
First / Last element Trivial Sorted or reverse-sorted arrays Teaching / simple demos only
Random element Simple (one extra swap) Extremely unlikely (probabilistic) Very common in production
Median-of-three Moderate (three comparisons) Specially constructed sequences Standard library implementations
Median-of-medians Complex Guaranteed O(n log n) Rarely — high constant factor

Average-Case Efficiency

Quicksort's most celebrated property is its average-case time complexity of O(n log n). To understand why, consider what happens when every partition step splits the array into two roughly equal halves. The recursion tree then has approximately log₂ n levels. At each level of the tree, the total amount of work done across all sub-arrays at that level is O(n) — every element participates in exactly one partition at each level. Multiplying the number of levels by the work per level gives O(n log n).

The partitions do not need to be perfectly equal to achieve this bound. Even a consistent 9-to-1 split — where one sub-array has 90% of the elements and the other has 10% — produces a recursion tree of depth O(log n), just with a slightly larger constant. As long as there is no systematic imbalance, the algorithm runs in O(n log n) time on average.

Compared to merge sort, which also runs in O(n log n) on average and in the worst case, quicksort is often faster in practice despite having the same asymptotic complexity. The reasons are concrete and worth understanding:

  • Cache locality: Quicksort accesses memory in a sequential, localized pattern during partitioning. Merge sort requires copying data between arrays, producing more cache misses on modern hardware.
  • Lower constant factors: The inner loop of quicksort's partition step is extremely tight — just comparisons and swaps within a contiguous block of memory. Merge sort's merge step involves more pointer management and auxiliary array writes.
  • In-place operation: Quicksort does not allocate O(n) auxiliary memory, reducing memory bandwidth pressure.

These practical advantages are why quicksort remains the default choice in many standard library implementations (often as a hybrid with insertion sort for small sub-arrays) even though merge sort offers a stronger worst-case guarantee.

Best and Worst-Case Performance

The best case for quicksort occurs when every pivot happens to be the exact median of its sub-array, splitting it into two equal halves at every level of recursion. This produces a perfectly balanced binary recursion tree of depth log₂ n, and the total work is O(n log n). In practice, perfect splits are rare, but splits that are close to equal are common enough that average-case performance approaches the best case.

The worst case arises when every partition step produces a maximally unbalanced split: one sub-array has zero elements and the other has all the remaining elements minus the pivot. This happens with a naive first-or-last-element pivot strategy when the input is:

  • Already sorted in ascending order
  • Already sorted in descending order
  • All identical elements (every element equals every other)

In this scenario, the recursion tree degenerates into a linear chain of depth n. The first call does O(n) work, the second does O(n−1), the third O(n−2), and so on. The total work is O(n) + O(n−1) + ... + O(1) = O(n²), making this as slow as bubble sort on the same input.

Randomized pivot selection transforms this worst case from a deterministic outcome for certain inputs into a statistically unlikely event. The probability of encountering O(n²) behavior with a random pivot on any specific input is astronomically small for large n, because it would require the random number generator to consistently pick the minimum or maximum element from every sub-array.

The space complexity of quicksort comes from the recursive call stack. Each call to the partition function uses O(1) space, but the stack stores one frame per level of recursion. In the average case with balanced partitions, the stack depth is O(log n). In the worst case, the stack depth is O(n), which can cause a stack overflow for large inputs. This can be mitigated by always recursing into the smaller sub-array first and using an iterative approach with an explicit stack for the larger sub-array — a technique called tail call optimization or Sedgewick's improvement, which guarantees O(log n) stack depth regardless of pivot quality.

Case Time Complexity Space Complexity (Stack) Condition
Best O(n log n) O(log n) Every pivot is the exact median
Average O(n log n) O(log n) Random or well-distributed input
Worst O(n²) O(n) Sorted / reverse-sorted / all-equal input with naive pivot

JavaScript Implementation

Implementing quicksort in JavaScript requires translating the conceptual steps — choose pivot, partition, recurse — into concrete index manipulation. The implementation below uses the Lomuto partition scheme with the last element of each sub-array as the pivot. It sorts the array in-place, meaning the original array is modified directly rather than returning a new one.

The base case of the recursion is when the sub-array has zero or one element (low >= high). An array of one element is already sorted by definition; an array of zero elements is empty. Both require no action and the function simply returns.

JavaScript's destructuring assignment provides a clean, idiomatic syntax for swapping two array elements without a temporary variable:

[arr[i], arr[j]] = [arr[j], arr[i]];

This is both readable and correct, though in performance-critical inner loops some developers prefer the traditional three-line swap with a temporary variable to avoid the overhead of creating a temporary array.

Here is a complete, annotated implementation:

function quicksort(arr, low = 0, high = arr.length - 1) {
  // Base case: sub-array of 0 or 1 element is already sorted
  if (low >= high) return;

  // Partition the array and get the pivot's final index
  const pivotIndex = partition(arr, low, high);

  // Recursively sort the sub-array to the left of the pivot
  quicksort(arr, low, pivotIndex - 1);

  // Recursively sort the sub-array to the right of the pivot
  quicksort(arr, pivotIndex + 1, high);
}

function partition(arr, low, high) {
  // Choose the last element as the pivot
  const pivot = arr[high];

  // i tracks the boundary: everything at index <= i is <= pivot
  let i = low - 1;

  for (let j = low; j < high; j++) {
    if (arr[j] <= pivot) {
      // Expand the "less than or equal" region and swap
      i++;
      [arr[i], arr[j]] = [arr[j], arr[i]];
    }
  }

  // Place the pivot in its correct sorted position
  [arr[i + 1], arr[high]] = [arr[high], arr[i + 1]];

  // Return the pivot's final index
  return i + 1;
}

// Example usage:
const data = [10, 7, 8, 9, 1, 5];
quicksort(data);
console.log(data); // [1, 5, 7, 8, 9, 10]

To trace through the execution on [10, 7, 8, 9, 1, 5] with pivot 5 (last element):

  • Scan: 10 > 5 (no swap), 7 > 5 (no swap), 8 > 5 (no swap), 9 > 5 (no swap), 1 ≤ 5 → swap positions 0 and 4: [1, 7, 8, 9, 10, 5]
  • Place pivot: swap positions 1 and 5: [1, 5, 8, 9, 10, 7]
  • Pivot 5 is now at index 1 — its final position.
  • Left sub-array [1] has one element, base case returns immediately.
  • Right sub-array [8, 9, 10, 7] is recursively sorted.

To upgrade this implementation with a random pivot, simply add a swap before calling partition:

function quicksortRandom(arr, low = 0, high = arr.length - 1) {
  if (low >= high) return;

  // Pick a random index in [low, high] and swap it to the end
  const randomIndex = Math.floor(Math.random() * (high - low + 1)) + low;
  [arr[randomIndex], arr[high]] = [arr[high], arr[randomIndex]];

  const pivotIndex = partition(arr, low, high);
  quicksortRandom(arr, low, pivotIndex - 1);
  quicksortRandom(arr, pivotIndex + 1, high);
}

This single addition — moving a random element to the pivot position before the standard Lomuto partition — provides strong probabilistic protection against worst-case inputs with negligible overhead. The rest of the partition logic remains unchanged, making this an easy and highly recommended upgrade from the naive fixed-pivot version.

For production use, it is also common to combine quicksort with insertion sort for small sub-arrays (typically those with fewer than 10–16 elements). Insertion sort has a small constant factor and performs very well on nearly-sorted small arrays, which is exactly what quicksort's recursive sub-arrays tend to look like near the base of the recursion tree. This hybrid approach is used in Java's Arrays.sort and many other standard library implementations.

NotesThe Lomuto vs. Hoare distinction is worth mentioning verbally in lecture; students often see only Lomuto in textbooks but Hoare in production code. Emphasize that the in-place nature of quicksort is a key practical advantage. The random pivot upgrade is a one-liner improvement that dramatically changes worst-case behavior and should be highlighted as a best practice.
