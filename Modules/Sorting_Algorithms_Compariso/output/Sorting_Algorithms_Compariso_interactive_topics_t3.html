Insertion Sort

▶

Insertion sort is one of the most intuitive sorting algorithms, closely mirroring the way a person might sort a hand of playing cards. You pick up one card at a time and slide it into its correct position among the cards you are already holding. The algorithm works by maintaining a sorted region on the left side of the array and an unsorted region on the right, progressively growing the sorted region until it encompasses the entire array. Despite its simplicity, insertion sort has genuine strengths that make it a practical choice in specific scenarios, and understanding it deeply provides a strong foundation for reasoning about more advanced sorting algorithms.

The algorithm begins with the sorted region containing only the very first element of the array — a single-element collection is trivially sorted. On each subsequent pass, it takes the first element of the unsorted region (called the key) and walks it leftward through the sorted region, comparing it against each element it encounters. Any sorted-region element that is greater than the key is shifted one position to the right, opening up a slot. This leftward walk continues until the algorithm finds an element that is less than or equal to the key, or until it runs off the left edge of the array. At that point the key is dropped into the vacated slot. After n − 1 passes for an array of n elements, the sorted region has swallowed the entire array and every element is in its correct position.

How Insertion Sort Works

It helps to think of the array as being divided by an invisible boundary that moves one step to the right after each pass. Everything to the left of the boundary is sorted relative to itself; everything to the right is waiting to be processed. The key insight is that the sorted region is not just "partially sorted" in some loose sense — it is fully sorted at every moment, which is why the leftward scan can stop as soon as it finds a smaller-or-equal element.

The three operations that drive insertion sort are:

  • Selection: Pick the first element of the unsorted region as the key.
  • Shifting: Move elements in the sorted region that are larger than the key one position to the right, overwriting the slot the shifted element just vacated.
  • Insertion: Place the key into the slot opened by the last shift (or into position 0 if all sorted elements were shifted).

Notice that "shifting" is not the same as "swapping." A swap exchanges two values and requires three assignments. Shifting overwrites a single slot, which is why insertion sort performs fewer write operations than bubble sort — an important practical advantage on real hardware where memory writes can be expensive.

Step-by-Step Example

Consider sorting the array [5, 2, 4, 6, 1, 3]. The table below traces the state of the array after each pass. The vertical bar | represents the boundary between the sorted (left) and unsorted (right) regions.

Pass Key Array State After Insertion What Happened
Start — [5 | 2, 4, 6, 1, 3] Sorted region is just index 0 (value 5).
1 2 [2, 5 | 4, 6, 1, 3] 2 < 5, so 5 shifts right; 2 is placed at index 0.
2 4 [2, 4, 5 | 6, 1, 3] 4 < 5, so 5 shifts right; 4 ≥ 2, so 4 is placed at index 1.
3 6 [2, 4, 5, 6 | 1, 3] 6 ≥ 5, no shifts needed; 6 stays at index 3.
4 1 [1, 2, 4, 5, 6 | 3] 1 is smaller than all sorted elements; all four shift right; 1 lands at index 0.
5 3 [1, 2, 3, 4, 5, 6] 3 < 4, 3 < 5, 3 < 6 (shifts); 3 ≥ 2, so 3 is placed at index 2.

Pass 3 is instructive: when the key is already larger than every element in the sorted region, the inner loop body never executes. The key effectively stays where it is. This is exactly what happens on every pass when the array is already sorted — leading to the best-case O(n) performance discussed later.

JavaScript Implementation

Below is a clean, well-commented JavaScript implementation of insertion sort. Read through the code and then the explanation that follows.

function insertionSort(arr) {
  // Outer loop: i is the index of the key (first element of unsorted region)
  // We start at 1 because a single element (index 0) is already "sorted"
  for (let i = 1; i < arr.length; i++) {

    // Save the current element before the inner loop overwrites positions
    let key = arr[i];

    // j will walk leftward through the sorted region
    let j = i - 1;

    // Shift elements that are greater than key one position to the right
    while (j >= 0 && arr[j] > key) {
      arr[j + 1] = arr[j]; // shift element right
      j--;                  // move one step left
    }

    // j+1 is the correct position for key:
    // either arr[j] <= key (so key goes just after j),
    // or j fell below 0 (so key goes at index 0, which is j+1 = 0)
    arr[j + 1] = key;
  }

  return arr;
}

// --- Example usage ---
let data = [5, 2, 4, 6, 1, 3];
console.log(insertionSort(data)); // [1, 2, 3, 4, 5, 6]

Let's walk through each structural decision:

  • Outer for loop starting at index 1: Index 0 is treated as the entire initial sorted region. Starting i at 1 means the very first key is arr[1], which is compared against the one-element sorted region [arr[0]]. This is correct and avoids an unnecessary iteration where nothing can be compared.
  • Saving the key before the inner loop: This is critical. The inner loop shifts elements rightward by writing arr[j + 1] = arr[j]. If i = 3 and we shift arr[2] into arr[3], the original value at index 3 would be lost unless we saved it in key first. The variable key holds the value we are trying to insert throughout the entire inner loop.
  • The while loop condition j >= 0 && arr[j] > key: The left boundary check (j >= 0) prevents reading before the start of the array. The comparison (arr[j] > key) uses strict greater-than, not greater-than-or-equal. This is what makes insertion sort stable: equal elements are not shifted past each other, so their original relative order is preserved.
  • Placing the key at arr[j + 1] after the loop: When the while loop exits, j points to the last element that did not need to be shifted (i.e., the last element ≤ key), or j is -1 if every sorted element was shifted. In either case, j + 1 is the correct insertion index. If j === -1, then j + 1 === 0, placing the key at the very beginning of the array.

It is worth noting that the sort operates in place: no second array is ever allocated. The only extra memory used is the single key variable and the loop indices, which is O(1) auxiliary space regardless of array size.

Time Complexity Analysis

Understanding the performance of insertion sort requires examining how many comparisons and shifts occur as a function of the input size n.

  • Best case — O(n): This occurs when the input array is already sorted in ascending order. On each of the n − 1 passes, the key is compared to exactly one element (the rightmost element of the sorted region) and found to be greater-than-or-equal, so the while loop exits immediately without any shifts. The total work is proportional to n comparisons, making the best case linear. This is a significant practical advantage: insertion sort degrades gracefully when given nearly sorted data.
  • Worst case — O(n²): This occurs when the array is sorted in reverse order (descending). On pass i, the key is smaller than all i elements in the sorted region, so all i of them must be shifted. Summing across all passes: 1 + 2 + 3 + … + (n − 1) = n(n − 1)/2 shifts and comparisons, which is O(n²). An array like [6, 5, 4, 3, 2, 1] triggers this worst case.
  • Average case — O(n²): For a randomly ordered array, on pass i the key is expected to be smaller than roughly half the sorted-region elements, requiring about i/2 shifts on average. Summing (1/2 + 2/2 + … + (n−1)/2) still yields O(n²). While the constant factor is roughly half that of the worst case, the quadratic growth dominates and average-case performance is still O(n²).
  • Space complexity — O(1): Insertion sort is an in-place algorithm. The single auxiliary variable key is the only extra storage required, making the space overhead constant regardless of the size of the input array. This contrasts with merge sort, for example, which requires O(n) auxiliary space for the temporary merge buffer.
Case Time Complexity Condition
Best O(n) Array is already sorted in ascending order
Average O(n²) Array is in random order
Worst O(n²) Array is sorted in reverse (descending) order
Space O(1) Always — only one temporary variable used

Stability and In-Place Properties

Two formal properties of sorting algorithms are especially important when choosing one for a real-world task: stability and in-place behavior.

A sorting algorithm is called stable if, whenever two elements have equal keys, they appear in the same relative order in the output as they did in the input. Consider sorting a list of student records first by grade and then by name. If the grade-sort is stable, students with the same grade will still be alphabetically ordered (from the prior name-sort) after the grade-sort completes. If it is unstable, that secondary ordering could be scrambled.

Insertion sort is inherently stable because the while loop condition uses strict greater-than (arr[j] > key). An element equal to the key is not shifted rightward; the loop stops and the key is inserted after its equal peer, preserving the original order. No equal elements ever pass each other. This guarantee holds without any special handling — it is a natural consequence of the algorithm's structure.

An algorithm is called in-place if it requires only a constant amount of auxiliary memory beyond the input itself — O(1) extra space. Insertion sort qualifies: the original array is rearranged by shifting elements within it, and the only extra variable needed is key. In memory-constrained environments (embedded systems, microcontrollers, environments where heap allocation is avoided), this property is highly desirable. An out-of-place algorithm like merge sort, by contrast, must allocate O(n) temporary storage to merge its subarrays.

Together, these two properties make insertion sort a strong candidate in situations where you need guaranteed stability without incurring any memory overhead — for instance, sorting a small linked list of records by one field while preserving existing ordering by another field.

Practical Use Cases and Comparisons

Given its O(n²) average and worst-case behavior, it might seem like insertion sort should be avoided in favor of O(n log n) algorithms like merge sort, quicksort, or heapsort. In practice, however, insertion sort fills several important niches.

  • Small arrays: Asymptotic analysis describes behavior as n grows large, but for small n the constant factors hidden inside Big-O notation matter enormously. Insertion sort has very low overhead — no recursive calls, no partitioning, no heap maintenance. For arrays of roughly 10–20 elements, benchmarks consistently show insertion sort outperforming merge sort or quicksort. This is why the standard libraries of many languages (including Python's Timsort and many implementations of C's qsort) switch to insertion sort for small subarrays within their divide-and-conquer algorithms.
  • Nearly sorted data: Because the inner loop makes comparisons and shifts only for elements that are out of place, insertion sort approaches O(n) performance when the input is "almost sorted" — for example, a sorted array with a handful of elements added to the end or a small number of elements that have drifted out of position. In such scenarios it dramatically outperforms algorithms with guaranteed O(n log n) behavior but higher constant costs.
  • Online sorting: Insertion sort is naturally suited to online algorithms — scenarios where elements arrive one at a time and must be incorporated into a sorted structure as they arrive, without the ability to see future elements. Because the sorted region grows incrementally and each new element is inserted into the correct position immediately, there is no need to wait for all elements before beginning. A real-time leaderboard that must stay sorted as new scores arrive is a classic example.
  • Comparison with bubble sort: Both insertion sort and bubble sort are simple O(n²) comparison-based sorts, but insertion sort is generally superior in practice. Bubble sort makes adjacent swaps (three assignments each) to move an element toward its destination; insertion sort shifts elements (one assignment each) and then performs a single insertion. For the same randomly ordered data, insertion sort typically performs fewer total data movements. Additionally, insertion sort's best case (already sorted) is O(n), while bubble sort's optimized best case is also O(n) but requires the extra overhead of a "swap occurred" flag. Without that flag, naïve bubble sort is O(n²) even on sorted data.
  • Comparison with selection sort: Selection sort always performs exactly n(n−1)/2 comparisons regardless of the input order (always O(n²)), while insertion sort can do as few as n−1 comparisons on a sorted array. Selection sort does have one advantage — it performs at most n − 1 swaps — but insertion sort's shifting approach is competitive in write-cost while also being adaptive (faster on easier inputs) and stable.

In summary, insertion sort is not merely a pedagogical toy. It is a carefully understood algorithm with well-defined strengths: O(n) best-case performance, O(1) space, guaranteed stability, natural fit for online processing, and superior practical speed on small or nearly-sorted datasets. Recognizing the conditions under which insertion sort is the right tool — rather than defaulting to more complex algorithms — is a mark of algorithmic maturity.
