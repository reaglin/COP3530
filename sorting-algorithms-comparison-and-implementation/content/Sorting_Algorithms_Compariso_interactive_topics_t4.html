Merge Sort

▶

Merge sort is one of the most elegant and reliable sorting algorithms ever devised. It belongs to a family of algorithms that solve problems by breaking them into smaller, more manageable pieces, solving each piece independently, and then combining the results. This general strategy — called divide and conquer — is the conceptual backbone of merge sort, and understanding it deeply unlocks not just this algorithm but an entire class of powerful computational techniques.

At its core, merge sort answers a deceptively simple question: if you already have two sorted lists, how quickly can you combine them into one sorted list? The answer turns out to be very fast — linear time. Merge sort exploits this by recursively reducing any unsorted list down to trivially sorted pieces (single elements), then building back up through repeated merging. The result is a guaranteed O(n log n) time complexity that never degrades, no matter what the input looks like.

Divide-and-Conquer Strategy

The divide-and-conquer paradigm has three phases: divide, conquer, and combine. Merge sort maps cleanly onto all three. In the divide phase, the algorithm splits the input array in half. In the conquer phase, it recursively sorts each half. In the combine phase, it merges the two sorted halves back together. This structure repeats at every level until the base case is reached.

The key insight is what constitutes the base case. An array containing zero or one element requires no sorting — it is already sorted by definition. A single number cannot be out of order with itself. This trivial fact is precisely what terminates the recursion and makes the whole process work. Every array, no matter how large or chaotically ordered, can be reduced to a collection of single-element arrays by repeated halving. Once you have that collection, you only need to merge them back together in the correct order.

To visualize this, consider the array [38, 27, 43, 3, 9, 82, 10]. The divide phase unfolds like a tree branching downward:

[38, 27, 43, 3, 9, 82, 10]
         /              \
  [38, 27, 43]       [3, 9, 82, 10]
    /      \            /        \
[38, 27]  [43]      [3, 9]    [82, 10]
  /    \              /   \      /    \
[38]  [27]          [3]   [9] [82]  [10]

At the bottom of that tree, every subarray is a single element. The algorithm then works its way back up, merging at each level. Merging [38] and [27] produces [27, 38]. That result merges with [43] to produce [27, 38, 43]. The right side undergoes the same process. Finally, [27, 38, 43] and [3, 9, 10, 82] are merged to produce the fully sorted array [3, 9, 10, 27, 38, 43, 82].

Recursive Structure of Merge Sort

The recursive structure of merge sort is what makes the divide-and-conquer strategy concrete and executable. The main mergeSort function does three things: it checks for the base case, it splits the array, and it recursively processes each half before delegating the actual combination work to a helper function.

The function accepts an array as its argument. If the array has a length of 0 or 1, the recursion has bottomed out — the subarray is already sorted, so the function simply returns it unchanged. This is critical: without a proper base case, the recursion would never terminate.

When the array has two or more elements, the function calculates the midpoint and uses Array.prototype.slice to divide the array into a left half and a right half. Both halves are then passed back into mergeSort recursively. Each recursive call goes through the same process — splitting, recursing, merging — until every branch of the call tree has resolved its base case. Only once both recursive calls return sorted arrays does the current call proceed to combine them.

This pattern — where a function calls itself on strictly smaller inputs and combines those results — is the hallmark of divide-and-conquer recursion. The call stack grows as the algorithm descends into smaller and smaller subarrays, and it unwinds as the merges propagate back upward. The depth of this call stack is logarithmic in the size of the input, because each split halves the problem size.

The Merge Helper Function

The merge helper is where the real sorting work happens. It receives two already-sorted arrays and produces a single sorted array containing all elements from both. The algorithm is elegant: it uses two pointers, one for each input array, each starting at index zero.

At every step, the merge function compares the current element from the left array against the current element from the right array. The smaller of the two is appended to the result array, and that pointer is advanced by one. This comparison-and-advance continues as long as both pointers are within bounds — that is, as long as neither input array has been fully consumed.

Once one array is exhausted, no more comparisons are needed. The remaining elements of the other array are guaranteed to be sorted and guaranteed to be larger than everything already added to the result (because both inputs were sorted to begin with). Those remaining elements are therefore appended to the result directly, in one operation, without any further comparisons.

Consider merging [27, 38, 43] and [3, 9, 10, 82]:

Step Left pointer Right pointer Comparison Element added Result so far
1 27 3 27 > 3 3 [3]
2 27 9 27 > 9 9 [3, 9]
3 27 10 27 > 10 10 [3, 9, 10]
4 27 82 27 < 82 27 [3, 9, 10, 27]
5 38 82 38 < 82 38 [3, 9, 10, 27, 38]
6 43 82 43 < 82 43 [3, 9, 10, 27, 38, 43]
7 (exhausted) 82 — 82 (appended directly) [3, 9, 10, 27, 38, 43, 82]

Because each element from both arrays is visited at most once, the merge function runs in O(n) time, where n is the total number of elements across both input arrays. This linear cost is what makes the overall algorithm so efficient when combined with the logarithmic depth of the recursion tree.

JavaScript Implementation

Below is a clean, complete JavaScript implementation of merge sort, separating the merge helper from the main recursive function for clarity.

// Merge helper: combines two sorted arrays into one sorted array
function merge(left, right) {
  const results = [];
  let i = 0; // pointer for left array
  let j = 0; // pointer for right array

  // Compare elements from both arrays and push the smaller one
  while (i < left.length && j < right.length) {
    if (left[i] <= right[j]) {
      results.push(left[i]);
      i++;
    } else {
      results.push(right[j]);
      j++;
    }
  }

  // Append any remaining elements from either array
  while (i < left.length) {
    results.push(left[i]);
    i++;
  }
  while (j < right.length) {
    results.push(right[j]);
    j++;
  }

  return results;
}

// Main merge sort function
function mergeSort(arr) {
  // Base case: arrays of length 0 or 1 are already sorted
  if (arr.length <= 1) return arr;

  // Find the midpoint and split the array
  const mid = Math.floor(arr.length / 2);
  const left = mergeSort(arr.slice(0, mid));
  const right = mergeSort(arr.slice(mid));

  // Merge the two sorted halves and return the result
  return merge(left, right);
}

// Example usage
console.log(mergeSort([38, 27, 43, 3, 9, 82, 10]));
// Output: [3, 9, 10, 27, 38, 43, 82]

Walk through what happens when mergeSort([38, 27, 43, 3]) is called:

  • The midpoint is Math.floor(4 / 2) = 2, so left = [38, 27] and right = [43, 3].
  • mergeSort([38, 27]) is called. Midpoint is 1. Left is [38], right is [27]. Both return immediately (base case). merge([38], [27]) returns [27, 38].
  • mergeSort([43, 3]) is called. Midpoint is 1. Left is [43], right is [3]. Both return immediately. merge([43], [3]) returns [3, 43].
  • Back at the top level, merge([27, 38], [3, 43]) is called and returns [3, 27, 38, 43].

Notice the use of <= rather than < in the comparison inside merge. When two elements are equal, preferring the left array element preserves stability — elements with equal keys maintain their original relative order from the input. This is an important property in many real-world sorting scenarios, such as sorting records by a secondary key while preserving a primary key's ordering.

Time Complexity: O(n log n)

Merge sort's time complexity can be understood by examining the recursion tree. At the top level, there is one array of size n. It splits into two arrays of size n/2. Each of those splits into two arrays of size n/4. This halving continues until all arrays reach size 1. The number of levels in this tree is log₂(n), because you can only halve n a logarithmic number of times before reaching 1.

At each level of the recursion tree, the merge operations collectively process every element exactly once. At the top level, one merge touches all n elements. At the second level, two merges each touch n/2 elements — still n total. At the third level, four merges each touch n/4 elements — still n total. This pattern holds at every level: the total work per level is always O(n).

Since there are O(log n) levels and each level does O(n) work, the total time complexity is O(n log n).

This is a crucial distinction from algorithms like insertion sort or bubble sort, which are O(n²) in the worst case, or even quicksort, which is O(n log n) on average but degrades to O(n²) when the pivot selection is consistently poor (e.g., always choosing the smallest or largest element on an already-sorted or reverse-sorted array). Merge sort has no such worst case. Whether the input is sorted, reverse-sorted, or entirely random, merge sort always performs exactly the same number of operations. This predictability is invaluable in systems where performance guarantees matter.

Algorithm Best Case Average Case Worst Case
Bubble Sort O(n) O(n²) O(n²)
Insertion Sort O(n) O(n²) O(n²)
Quicksort O(n log n) O(n log n) O(n²)
Merge Sort O(n log n) O(n log n) O(n log n)
Heapsort O(n log n) O(n log n) O(n log n)

Space Complexity and Trade-offs

Merge sort's primary drawback is its memory usage. Because the merge step combines two subarrays into a brand-new array rather than rearranging elements in place, it requires O(n) auxiliary space in addition to the space occupied by the original input. At any given moment during the algorithm, newly created subarrays from the merge operations are held in memory. The largest of these is at the final merge, which creates an array of size n. Across the entire recursion tree, the peak memory usage is O(n).

Additionally, because merge sort is recursive, each active function call occupies a frame on the call stack. The maximum depth of the recursion is log₂(n) (the height of the recursion tree), so the call stack contributes an additional O(log n) space. For most practical purposes, this is negligible compared to the O(n) heap allocation, but it is worth noting — particularly in environments with limited stack depth.

Contrast this with insertion sort, which sorts in place using only a constant amount of extra memory (O(1) space), or heapsort, which also sorts in place in O(n log n) time. For memory-constrained environments, those algorithms may be preferable despite other trade-offs.

However, merge sort's memory usage is often entirely acceptable, and in some contexts it is the only sensible choice:

  • Linked lists: Merge sort is the natural choice for sorting linked lists because splitting a linked list at its midpoint and merging two linked lists are both efficient operations requiring no extra memory. Quicksort, by contrast, struggles with linked lists because random access (needed for efficient pivot selection and partitioning) is O(n) rather than O(1).
  • External sorting: When data is too large to fit in RAM and must be sorted across disk files or tape drives, merge sort's sequential access pattern is ideal. Data is read and written in large sequential chunks rather than accessed randomly, which matches the performance characteristics of disk I/O. Database systems and file-sorting utilities frequently use variants of external merge sort for exactly this reason.
  • Stable sorting requirements: As noted earlier, merge sort is naturally stable. When the relative order of equal elements must be preserved — as in multi-column spreadsheet sorting or database query results — merge sort is an excellent choice.
  • Parallelism: The independent nature of the two recursive halves makes merge sort highly amenable to parallel execution. Both halves can be sorted simultaneously on different processors, and the merge step can also be parallelized. This makes merge sort the foundation of many parallel and distributed sorting algorithms.

In summary, merge sort is a foundational algorithm that every serious programmer should understand deeply. Its consistent O(n log n) performance, its stability, its suitability for linked lists and external data, and its clean recursive structure make it not just a classroom exercise but a genuinely practical and widely deployed tool. The trade-off in auxiliary memory is real, but in the vast majority of applications it is a trade-off well worth making.

NotesThe JavaScript implementation uses separate while loops to drain remaining elements from either array after one is exhausted, which is clearer for learners than a single concatenation. Instructors may wish to demonstrate the spread-operator shorthand (e.g., `...left.slice(i)`) as an alternative but should note that the explicit loop version makes the logic most transparent.
